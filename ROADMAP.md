# Oblivion Bot — Roadmap

Telegram-бот, первый продукт экосистемы Oblivion. Использует новую Telegram Business API (Bot API 9.5+, март 2026) для чтения входящих сообщений и ответов от имени пользователя через LLM. Долгосрочно — фронт мульти-канального приватного AI-ассистента, работающего на конфиденциальных вычислениях TON Cocoon.

---

## Vision

Бот, который отвечает за тебя — но **только так, как ответил бы ты**, и **никогда никому не передаёт содержание твоих диалогов** в открытом виде. Лозунг продукта: «An AI that forgets you on purpose».

Это не автоответчик. Это персональный AI-агент: отвечает в диалогах, решает задачи через скиллы (календарь, напоминания, заметки, поиск — добавляются по мере роста), помнит факты о людях и контексте, даёт рекомендации, ведёт твой список дел. По сути — мега-ассистент, который заменяет десяток сервисов и при этом не сливает данные.

Четыре уровня значения:
1. **Утилита** — освобождает время на бытовых ответах, помнит то, что ты забываешь, выполняет рутину через скиллы.
2. **Приватность** — не словом, а архитектурой. Локальное хранение по умолчанию, шифрование на диске, маскирование PII перед LLM-вызовом, опциональный переход на TEE-инференцию через Cocoon. См. фазу «Privacy by architecture».
3. **Персонализация** — каждый пользователь настраивает бота под себя через Mini App (тон, скиллы, память, правила приватности). Без интерфейса бот — это терминал; с Mini App — это твой ассистент.
4. **Принцип** — выбор модели, степени фильтрации, того, что бот помнит и кому отвечает, остаётся за пользователем, не за платформой.

## Архитектура

### База форка
- **Источник**: `github.com/AbhiTheModder/tgbot-business-ai`
- **Стек**: Python 3.10+, aiogram 3.x (нативная Bot API библиотека), официальная Bot API 9.5+
- **Расположение исходников**: `oblivion-bot/src/` (clone в эту подпапку, чтобы не смешивать с проектной документацией)

### Stack-решения, принятые
- **Язык**: Python (оставляем как в форке — переписывать смысла нет)
- **Telegram-клиент**: aiogram 3.x. Форк изначально использовал Pyrofork (MTProto), но это лишняя сложность: Business updates прилетают по Bot API HTTPS с версии 9.5. Aiogram — нативный Bot API клиент, async, без session-файлов и MTProto-криптографии. `api_id`/`api_hash` не нужны.
- **Telegram API**: только официальный Bot API. **Никаких MTProto userbot-паттернов** (юридически серая зона + лишние credentials).
- **LLM по умолчанию для MVP**: DeepSeek (cheapest reliable) или Together (Llama-3.3, Qwen-2.5). Конфигурируется через `.env`.
- **Память**: per-user история диалогов в SQLite (локальный файл, шифрование на следующих фазах).
- **Конфиг**: всё через `.env` файл, никаких хардкоженных ключей.

### Связь с другими частями экосистемы Oblivion
- **oblivion-core** (соседняя папка, форк OpenCode) — десктоп-агент для разработчиков. Параллельный продукт, общий бренд и LLM-бэкенд.
- **Cocoon adapter** (фаза 5) — общий модуль, который оба продукта используют как inference backend. Пишется отдельно, плагином.
- **Дизайн-код Lethean** — голос бота должен соответствовать (см. секцию ниже).

### Архитектурные опоры (поверх форка)

Эти модули добавляются поверх tgbot-business-ai и определяют отличие Oblivion от типового AI-автоответчика. Слои строго разделены — ядро не лезет в БД напрямую, скиллы не лезут в LLM напрямую, всё проходит через явные интерфейсы.

- **Skills engine** — модульная система навыков. Каждый скилл = отдельный плагин с явным манифестом (`name`, `description`, `input_schema`, `output_schema`, `required_permissions`). LLM выбирает скилл через function-calling по описанию. Скиллы добавляются без правки ядра — кинул файл в `skills/`, перезапустил, готово. См. фазу 2.5.
- **Memory layer** — структурированная память поверх SQLite. Не просто история диалогов, а: `facts` (что бот знает о людях и темах), `tasks` (твои дела с дедлайнами и статусами), `preferences` (как ты предпочитаешь общаться). Память доступна как контекст для LLM и как объекты, которыми управляют скиллы.
- **Privacy core** — единая точка прохода для всех операций с пользовательскими данными. Отвечает за шифрование на диске, маскирование PII перед отправкой в LLM, audit log, политики хранения. Никакой код не пишет в БД и не вызывает LLM, минуя этот слой.
- **Mini App (TWA)** — Telegram Web App для персонализации. Управление настройками, скиллами, памятью, приватностью через интерфейс, а не команды. Подключается к боту через тот же backend. См. фазу 4.

## Voice & Tone (КРИТИЧНО для системного промпта)

Бот говорит коротко, по делу, без подобострастия. Это не «AI ассистент с восклицательными знаками».

**Правила:**
- Lowercase по умолчанию, заглавные только для важных предупреждений
- Никаких «Конечно!», «С удовольствием!», «Буду рад помочь!»
- Ошибки сообщаем, не извиняемся за них
- Если не понятно — переспрашиваем одной фразой, не структурой из 3 уточняющих вопросов
- Длинные ответы только если реально нужны — обычно 1-3 предложения
- Можно использовать «—» вместо длинных пояснений
- Эмодзи — никогда, если только пользователь сам не общается смайлами

**Bad** (стандартный SaaS): «Привет! Конечно, я с радостью помогу тебе с этим вопросом! Давай разберёмся вместе...»
**Good** (Oblivion): «давай. что нужно?»

**Bad**: «К сожалению, мне не удалось обработать ваш запрос. Пожалуйста, попробуйте позже.»
**Good**: «не дошло. повтори.»

## Phase 0 — Setup

Цель: рабочий бот, который отвечает на твоё первое тестовое сообщение через LLM.

- [ ] Склонировать `tgbot-business-ai` в `oblivion-bot/src/` (`git clone https://github.com/AbhiTheModder/tgbot-business-ai.git src`)
- [ ] Прочитать `src/README.md`, понять команды запуска
- [ ] Прочитать `src/bot.py` — точка входа, обработчик Business updates, LLM-вызов
- [ ] Создать venv: `python3 -m venv .venv && source .venv/bin/activate`
- [ ] Установить зависимости: `pip install -r src/requirements.txt`
- [ ] Создать бота через `@BotFather` в Telegram, сохранить токен
- [ ] В Telegram → Settings → Business → Chatbots добавить токен. Включить пермишены: «Reply to messages», «Read messages»
- [ ] Получить API-ключ для LLM (OpenRouter через `openrouter.ai` — единый ключ под любые модели, можно DeepSeek/Qwen/Llama)
- [ ] Создать `oblivion-bot/.env` с токенами (см. шаблон ниже)
- [ ] Запустить бота: `python src/main.py` (или эквивалент)
- [ ] С другого Telegram-аккаунта написать тебе — убедиться что бот отвечает

### Шаблон `.env`

```env
# Telegram Bot (Bot API через aiogram — api_id/api_hash НЕ нужны)
BOT_TOKEN=

# LLM Provider — OpenRouter (OpenAI-совместимый, единый ключ под любые модели)
LLM_PROVIDER=openrouter
LLM_API_KEY=
LLM_MODEL=deepseek/deepseek-chat-v3.1
LLM_BASE_URL=https://openrouter.ai/api/v1

# Bot behavior
BOT_OWNER_ID=  # твой Telegram user ID, узнать у @userinfobot
LOG_LEVEL=INFO
```

**Done when**: тебе пришло сообщение в личку, бот ответил через LLM, ответ виден в Telegram. Никаких ошибок в логе.

## Phase 1 — MVP

Цель: бот ведёт себя как Oblivion, не как стандартный AI-ассистент.

- [ ] Заменить дефолтный системный промпт на Oblivion voice & tone (см. секцию выше). Промпт вынести в `oblivion-bot/src/prompts/system.txt` или `system_oblivion.md`
- [ ] Команды управления для владельца бота (только `BOT_OWNER_ID`):
  - `/pause` — приостановить автоответы
  - `/resume` — возобновить
  - `/forget @username` — стереть историю с конкретным собеседником
  - `/forget all` — wipe всей памяти
  - `/context @username` — показать что бот помнит про собеседника
  - `/mode work|casual|formal` — переключить тональность ответов
- [ ] История по каждому собеседнику в SQLite (если в форке нет — добавить). Схема: `user_id`, `message_text`, `role (user|assistant)`, `timestamp`
- [ ] Whitelist/blacklist собеседников: бот отвечает только определённым / не отвечает определённым. Конфиг в БД, управляется командами
- [ ] Длинные сообщения — разбивать на части (Telegram limit 4096 символов)
- [ ] Логи вызовов LLM в локальный файл `oblivion-bot/logs/`. Команда `/wipe-logs` для полного удаления
- [ ] Graceful shutdown — при SIGINT доотвечать на текущее, потом выйти

**Done when**: бот ведёт диалог так, как написано в voice & tone, ты можешь управлять им через команды, история работает.

## Phase 2 — Intelligence & Memory

Цель: бот не просто отвечает — он отвечает **как ты**, помнит факты, ведёт твои задачи.

- [ ] Анализ тональности входящего сообщения (formal/casual/playful), бот подстраивает свой ответ
- [ ] **Memory layer (фундамент)**:
  - Таблица `facts`: `subject`, `key`, `value`, `source_chat`, `created_at`, `confidence`. Пример: «Иван — партнёр по проекту X», «Маша из Питера, любит вино».
  - Таблица `tasks`: `title`, `due_at`, `status`, `created_from_chat`, `priority`. Создаются автоматически при детекции дедлайна в диалоге, явной командой или вызовом скилла.
  - Таблица `preferences`: `key`, `value`. Заполняется из Mini App и через явные команды.
  - LLM получает релевантные факты как часть контекста (RAG-light: keyword match по `subject` на старте, векторный поиск — позже, если будет нужно).
- [ ] Detection транзакционных диалогов (просьбы, договорённости, дедлайны) — авто-создание `task` со статусом `needs_review`, владелец подтверждает в Mini App или командой
- [ ] Stop-темы — список тем, на которые бот не отвечает автоматически (финансы, здоровье, эмоции). Помечает чат для ручного ответа.
- [ ] Streaming ответов через `sendMessageDraft` (показывать что бот «печатает» как живой человек)
- [ ] Опциональная «человечность» — рандомная задержка перед ответом 3-15 секунд, типизирующий индикатор

**Done when**: бот помнит факты о собеседниках, автоматически создаёт задачи из диалогов, проходит turing-test для коротких обменов (3-5 сообщений).

## Phase 2.5 — Skills Engine

Цель: бот не только отвечает, но и **действует** — через модульные навыки.

- [ ] Скилл-манифест: Python class с обязательными полями `name`, `description`, `input_schema`, `output_schema`, `required_permissions`, метод `handle(args, ctx) -> result`. Описание — то, что LLM видит и по чему решает вызывать.
- [ ] Скилл-роутер: LLM получает список доступных скиллов через function-calling, ядро исполняет вызов, результат скилла возвращается в контекст ответа.
- [ ] Базовые скиллы для бутстрапа:
  - `remember(subject, key, value)` — записать факт в `facts`
  - `recall(query)` — вспомнить факты
  - `add_task(title, due_at?, priority?)` — создать задачу
  - `list_tasks(filter?)` — показать задачи (open / today / overdue / all)
  - `complete_task(id)` — закрыть задачу
  - `forget(target)` — стереть память по теме / собеседнику
  - `search_web(query)` — поиск в интернете (опционально, требует разрешения)
- [ ] Каждый скилл явно объявляет, какие данные пользователя он трогает (read/write по `facts` / `tasks` / `preferences` / `chat_history` / `external_network`). Privacy core блокирует скилл, если разрешение не выдано.
- [ ] Логирование вызовов скиллов в audit log: какой скилл, аргументы (с маскировкой), результат (с маскировкой), кто инициировал (LLM auto / user explicit / Mini App).
- [ ] Конфиг скиллов: какие включены/выключены, управляется через Mini App.

**Done when**: можно написать новый скилл одним файлом, кинуть в `skills/`, перезапустить — бот видит, вызывает, логирует. Без правки ядра.

## Phase 3 — Privacy by Architecture

Цель: приватность — не маркетинговое слово, а свойство системы. Любой пользователь может прочитать код и убедиться сам.

### Принципы

- **Local-first**. Все данные пользователя по умолчанию живут на его машине / его сервере. Облако — опт-ин для конкретных функций.
- **Минимизация запросов к LLM**. В контекст попадает только то, что нужно для текущего ответа. PII маскируется (имена → `<contact_n>`, телефоны/email/адреса → плейсхолдеры) перед отправкой, восстанавливается после.
- **Forget значит forget**. `/forget` физически удаляет строки из БД (включая `VACUUM` для SQLite), а не помечает флагом.
- **Прозрачность**. Audit log append-only, доступен пользователю в любой момент через Mini App.
- **No silent exfiltration**. Каждый сетевой вызов из бота — задокументирован. Никаких метрик, телеметрии, «check for updates».

### Задачи

- [ ] Privacy core — единый модуль, через который проходят все операции с пользовательскими данными (БД, LLM, сеть)
- [ ] Шифрование БД (SQLCipher, ключ в Keychain / Linux keyring / passphrase для self-hosted)
- [ ] Маскировка PII перед отправкой в LLM (через privacy core, обязательная — нельзя отключить «для удобства»)
- [ ] Опциональный disclaimer-режим: при первом ответе боту собеседнику отправляется одно сообщение «вы общаетесь с AI-ассистентом». По умолчанию выключен, переключатель в Mini App
- [ ] Auto-purge старых диалогов через N дней (конфигурируется через Mini App, по умолчанию 30)
- [ ] Per-chat настройки приватности: для конкретного чата отключить запоминание, отключить скиллы с внешними вызовами, поднять/опустить уровень маскировки
- [ ] Audit log: timestamp, action (reply / skill_call / memory_write / memory_read / forget / config_change), target, summary. Append-only файл, не редактируется ботом
- [ ] Команда `/audit [N]` — последние N записей; полная история через Mini App
- [ ] Кнопка «Forget everything» в Mini App — wipe всей памяти + VACUUM + ротация ключа шифрования. С двойным подтверждением
- [ ] Документ `PRIVACY.md`: что хранится, где, какие сетевые вызовы делает бот. Без юридического канцелярита, в стиле voice & tone
- [ ] Документ `THREAT_MODEL.md`: явно — от чего защищаемся, от чего нет. Пример: защищаемся от LLM-провайдера, видящего полную переписку; не защищаемся от компрометации устройства пользователя
- [ ] Reproducible builds для Docker-образа (опц., можно отнести в фазу 4) — self-hosted бинарь = hosted бинарь, побайтово

**Done when**: продукт можно показать юристу, журналисту, параноику — и `PRIVACY.md` + `THREAT_MODEL.md` отвечают на все их вопросы. Зацепиться не за что.

## Phase 4 — Distribution & Mini App

Цель: другие люди могут использовать бота. Управление — через Mini App, не через команды (команды остаются как fallback и для CLI-любителей).

### Mini App (TWA) — основной интерфейс персонализации

Telegram Web App, открывается по кнопке в боте или ссылке. Технически — отдельный фронт, общающийся с тем же backend.

- [ ] Стек: TypeScript + React + Telegram Web App SDK. Деплой статикой (один HTML/JS bundle), backend — тот же что у бота, эндпоинты `/api/*` под Telegram `initData` auth
- [ ] Авторизация через Telegram `initData` (HMAC-проверка) — никаких отдельных аккаунтов, паролей, OAuth
- [ ] Экраны:
  - **Главный**: статус (online / paused), последние ответы, последние созданные задачи, быстрые тумблеры
  - **Persona**: voice & tone слайдер, режим (work / casual / formal), системные инструкции бота — текстовое поле, можно подправить
  - **Memory → Facts**: список фактов, поиск, правка, удаление
  - **Memory → Tasks**: задачи (open / today / overdue / done), отметить выполненными, добавить вручную
  - **Контакты**: whitelist / blacklist, per-chat настройки (отвечать или нет, запоминать или нет, какой тон)
  - **Skills**: каталог установленных скиллов, тумблеры on/off, разрешения для каждого (read/write по типам данных, external network)
  - **Privacy**: auto-purge, disclaimer, уровень маскировки PII, «Forget everything» с двойным подтверждением
  - **Audit log**: последние действия, фильтры, поиск
  - **LLM**: провайдер, модель, API key (BYOK), Cocoon-режим (когда будет фаза 5)
- [ ] Дизайн строго по `oblivion-design-system.html` (Lethean direction). Lowercase, минимализм, моноширинные акценты
- [ ] Все мутации (изменения настроек, удаления) — с подтверждением. Wipe-операции — двойным
- [ ] Локализация: RU + EN на старте
- [ ] Mini App работает только когда открыта пользователем — никаких фоновых запросов, никакого хранения данных в localStorage кроме сессионных токенов

### Дистрибуция

- [ ] Hosted версия: бот + Mini App развёрнуты на нашем сервере (VPS), пользователь только привязывает к своему аккаунту
- [ ] Self-hosted: docker-compose + готовые образы на ghcr.io. Один файл `docker-compose.yml`, поднимается `docker compose up -d`. Mini App в том же compose, отдельный контейнер.
- [ ] Bring-your-own-key для LLM
- [ ] Telegram Stars или TON оплата для hosted версии (фаза 5 закрывает TON)
- [ ] Лендинг `project-oblivion` (домен куплен, дизайн готов в `OBLIVION-DESIGN-SYSTEM.html`)

**Done when**: посторонний человек по инструкции с лендинга поднимает бота + Mini App за 10 минут и настраивает всё через интерфейс, не трогая `.env` после первичного запуска.

## Открытые технические вопросы

Решаются по ходу, но фиксируем чтобы не забыть:

1. **Где хостить hosted-версию.** Hetzner / Fly.io / собственный bare-metal? Влияет на cost-per-user.
2. **Какой LLM по умолчанию для MVP.** Кандидаты: DeepSeek (дёшево, англ), Qwen-2.5 через Together (мультиязычно, дёшево), Qwen3.5-21B-Heretic-Uncensored (наша «фишка», но сложнее хостить).
3. **Streaming vs batch ответы.** Bot API 9.5 даёт `sendMessageDraft` — стоит ли тратить время на streaming в MVP или достаточно batch.
4. **Тайминг ответов.** Мгновенно (выглядит как бот) vs human-like delay 5-30 сек (выглядит как ты, но ухудшает UX). Сделать настраиваемым.
5. **Как обрабатывать голосовые сообщения.** Whisper для транскрипции? Игнорировать? Помечать как `[needs review]`?
6. **Группы vs личка.** В MVP — только личка. Группы это отдельная архитектурная сложность (несколько собеседников, разный контекст).
7. **Multi-user hosted версия — изоляция данных.** Каждый пользователь в своей схеме SQLite, или один общий с `user_id` колонкой? Влияет на безопасность при компрометации.
8. **Skill manifest формат.** Python class (типизированно, удобно для IDE) vs YAML + handler (декларативно, проще для не-Python скиллов). Старт — Python class, но интерфейс должен оставлять возможность YAML позже.
9. **Mini App — деплой.** Та же машина что и бот (проще) или отдельный CDN/Pages (быстрее, но больше движущихся частей). В self-hosted — точно та же машина.
10. **Mini App auth.** Достаточно ли Telegram `initData` HMAC, или для критичных операций (wipe, изменение системного промпта) добавить ещё один шаг подтверждения через bot-команду?
11. **PII-маскировка — где порог.** Маскируем только имена / телефоны / email, или ещё топонимы, даты? Чем агрессивнее — тем хуже качество ответов LLM. Нужна настройка уровня в Mini App.
12. **Vector memory.** SQLite + keyword match на старте достаточно, но рано или поздно понадобится семантический поиск. sqlite-vss? отдельный chromadb? Решаем при первой реальной потребности, не заранее.

## Принципы разработки

Вне зависимости от фазы:

- **Privacy by default**, не opt-in. Если приходится выбирать между удобством и приватностью — приватность.
- **Никаких analytics, никакого telemetry** в код, который пользователь хостит у себя.
- **Никаких external dependencies**, которые делают сетевые вызовы помимо тех, что мы явно объявили (Telegram API + LLM endpoint + Cocoon — это всё).
- **Кодстайл**: black + ruff. Никаких религиозных войн, тулинг решает.
- **Тесты**: покрытие не цель, но critical paths (Business update handler, LLM dispatcher, БД) обязательно покрыты.
- **Коммиты**: conventional commits (`feat:`, `fix:`, `chore:`, etc).
- **Брендинг**: ничего не «AI Assistant», везде «Oblivion». README должен быть написан в voice & tone бота — то есть короткий, без маркетинга.

## Что НЕ делать

- Не переписывать бота с нуля «на TypeScript для консистентности с oblivion-core» — Python-форк работает, переписывание = месяц потраченного времени без улучшения продукта.
- Не добавлять «AI features» которые мы не используем сами. Каждая фича должна решать конкретную боль из реального использования.
- Не хардкодить credentials. Никогда. Даже в gitignore'нутых файлах.
- Не делать «smart» дефолтные настройки приватности (например, авто-включать запоминание для «лучшего опыта»). Дефолт = максимальная приватность, пользователь явно расширяет.
- Не строить multi-tenant архитектуру в MVP. Один пользователь = одна инсталляция. Multi-tenant — фаза 4+.
- Не делать скиллы, которые ходят в сеть, без явной декларации в манифесте и без галочки в Mini App.
- Не делать «smart дефолты» в Mini App (типа «давайте поставим тон friendly, чтоб приятнее»). Все дефолты нейтральные/строгие, пользователь явно настраивает.
- Не кешировать пользовательские данные в Mini App дольше сессии. Открыл → подтянул из backend, закрыл → выгрузил.
- Не строить векторную память «на всякий случай». SQLite + keyword достаточно, пока сам не упрёмся в потолок.
- Не писать собственный auth в Mini App. Telegram `initData` — единственный источник идентичности.
